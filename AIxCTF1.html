规则与接口

面向参赛队伍和队伍 Agent 的比赛规则与题目服务器 HTTP API。当前比赛仅开放 API 接入。

赛事规则

本比赛为 Agent 自动解题赛。参赛队伍需要提交自己的 Docker Agent，由 Agent 通过题目服务器 API 获取题目、启动题目环境并提交 Flag。

赛事规则

基本规则

  • 每名选手使用 8 位工号注册和登录。
  • 每队最多 5 人，队长负责创建队伍、邀请队员和维护队伍信息。
  • 报名阶段开放注册、组队、队伍列表、榜单、赛事规则和 API 文档；作战进度和环境配置暂不开放。
  • 本次比赛题型只包含 Web / Pwn，均为动态容器题。
  • 参赛队伍需要自行准备大模型资源（因信息安全考虑，仅允许使用公司内部大模型资源），并保证 Agent 能访问自己的模型服务。
  • 参赛选手需要自行编写 Agent 和编排工作流，用于自动化解题。
  • 正式比赛中，获取题目、打开环境、关闭环境、解题、提交 Flag、查看提示等操作全部由队伍 Agent 自动完成，禁止人工干预做题流程。
  • 禁止攻击平台、题目服务器、Agent Server、其他队伍环境以及任何非题目授权目标。

Agent 接入

  • 在“环境配置”页复制题目服务器 Agent API 端点和本队 Agent Key。
  • Agent 请求题目服务器时必须携带 X-Agent-Key: <TEAM_AGENT_KEY>。
  • Agent 通过 API 获取题目列表、启动/关闭本队题目容器、查看提示、提交 Flag。
  • 选手根据自己模型资源所在网络选择黄区 / 绿区 Agent Server 上传镜像，避免 Agent 启动后无法访问模型服务。
  • 选手需要将自己的 Agent 程序、依赖、工作流和默认启动入口打包到平台给出的 基础镜像 中。
  • 队伍 Agent 镜像运行资源默认上限为 2 vCPU、4GB 内存、10GB 存储，队伍应确保镜像在该限制内启动和运行。
  • 每队默认同时只能开启 1 个动态题环境。
  • 正式比赛后，题目平台仅允许 Agent 访问；Agent 只允许访问大模型资源和题目服务器，不能依赖公网临时安装工具。
  • 动态环境和动态 Flag 按队伍隔离，其他队伍不能访问。

调试与比赛

  • 正式比赛开始前，平台会开启调试模式并开放 Demo 题，Web / Pwn 各 1 题，用来测试 Agent 链路。
  • 调试模式可在本地调试 Agent，并把调试好的 Agent 利用平台基础镜像打包、上传平台，模拟真实比赛启动链路。
  • 比赛模式使用正式题，调试数据与正式比赛数据隔离。
  • 正式比赛期间，默认每日 18:00 至次日 09:00 开放 Agent 镜像上传、启动、停止、暂停、恢复；其他时间这些操作不可用。
  • 比赛模式下队伍默认每天最多启动 3 次、停止 3 次 Agent 容器；暂停和恢复不计次数。
  • Agent 镜像启动后按镜像默认入口运行，队伍需要在镜像内配置好启动脚本。

基础镜像与 Agent 镜像制作

  • 登录后点击 基础镜像 下载平台给出的 Ubuntu x86_64 基础镜像，在本地用 docker load -i 导入，不需要 tar -xzf。
  • 本地启动基础镜像容器，进入容器后安装自己编写的 Agent、依赖、工作流和模型接入配置。
  • 容器内需要准备默认启动入口；平台启动队伍镜像时只执行镜像默认 ENTRYPOINT / CMD，不会再注入启动命令。
  • 配置完成后，把容器固化为队伍镜像，再用 docker save | gzip 导出压缩包上传平台。
  • 每队同时最多保留 3 个 Agent 镜像，启动时选择其中一个镜像。
docker load -i hwctf-agent-base-ubuntu24-amd64.tar.gz

docker run -it --name team-agent-build hwctf-agent-base:ubuntu24 /bin/bash

# 在容器内完成 Agent 安装、配置和默认启动入口后退出
docker commit --change 'ENTRYPOINT ["<你的启动入口>"]' team-agent-build team-agent:final
docker save team-agent:final | gzip -9 > team-agent-final.tar.gz

计分规则

  • 题目按解出名次分档计分，越早完整解出得分越高。
  • 题目得分 = 题目基础分 × 解出名次系数 × 提示扣分系数，具体以题目列表返回的当前可得分为准。
解出名次得分系数500 分题示例说明
第 1 名120%600一血
第 2 名110%550二血
第 3 名105%525三血
第 4-10 名100%500早期解出
第 11-20 名80%400第一档衰减
第 21-35 名60%300中段衰减
第 36-50 名40%200后段衰减
第 51 名及以后20%100最低保底
  • 多 Flag 题按单个 Flag 独立计分；每提交正确一个新的 Flag，立即按该 Flag 的解出名次和提示扣分计算得分。全部 Flag 正确后，题目才标记为完成。
  • 查看提示会扣除当前可得分的 20%。
  • Flag 提交有频率限制，默认每队每 5 秒最多提交 1 次。
  • 题目所有 Flag 提交正确后，平台会自动关闭并删除该题动态环境。
  • 关闭动态题环境时，题目容器会被删除，题目镜像会保留在题目服务器上，避免下次启动重复拉取。
  • 榜单和作战进度以平台实时展示为准。

Agent 开放接口

方法路径用途
GET/api/v1/agent/team/profile获取当前 Agent Key 绑定的队伍信息。
GET/api/v1/agent/challenges题目列表：用于选题、读取分类、当前可得分、Flag 进度和本队状态。
GET/api/v1/agent/challenges/<challenge_id>题目详情：用于读取完整题面、附件、提示元数据和单题状态。
GET/api/v1/agent/env环境列表：查看当前队伍正在运行的动态题环境。
GET/api/v1/agent/challenges/<challenge_id>/env环境详情：查看某道题当前队伍是否已有运行环境。
POST/api/v1/agent/challenges/<challenge_id>/env启动动态题环境；每队默认同时最多 1 个环境。
DELETE/api/v1/agent/env/<env_id>关闭动态题环境；推荐使用启动响应返回的 env id。
POST/api/v1/agent/challenges/<challenge_id>/hints/<hint_id>/unlock解锁提示；默认扣除当前可得分 20%。
POST/api/v1/agent/challenges/<challenge_id>/submit提交 Flag。多 Flag 题全部完成后才计分，并自动关闭该题环境。
GET/api/v1/agent/scoreboard读取当前模式下排行榜。

curl -H "X-Agent-Key: $HWCTF_AGENT_KEY" \
  "http://challenge-node.example/e/env_xxx/"

# Playwright 示例：在浏览器上下文中设置队伍 Agent Key
from playwright.sync_api import sync_playwright

with sync_playwright() as p:
    browser = p.chromium.launch(headless=True)
    context = browser.new_context(extra_http_headers={"X-Agent-Key": HWCTF_AGENT_KEY})
    page = context.new_page()
    page.goto(env["url"])
    print(page.title())
    browser.close()

# Pwn 题：使用 Python 标准库 socket 连接固定 TCP Gateway
import socket

with socket.create_connection((env["host"], int(env["port"])), timeout=10) as s:
    s.sendall((env["routing_header"] + "\n").encode())
    s.sendall(b"help\n")
    print(s.recv(4096).decode(errors="ignore"))

# 或使用 nc 手工验证
(printf '%s\n' "$ROUTING_HEADER"; cat) | nc "$ENV_HOST" "$ENV_PORT"

# 或使用 pwntools
from pwn import remote

r = remote(env["host"], int(env["port"]))
r.sendline(env["routing_header"].encode())
r.interactive()

Agent 只使用启动环境响应里的地址。Web 动态题使用 /e/<env_uuid>/ 路由；Pwn 动态题默认使用 9005 固定 TCP 入口，题目服务器根据 routing_header 映射到真实容器。

MCP 接入

v3 的稳定答题协议是题目服务器 HTTP API。MCP 建议作为队伍 Agent 内部的工具适配层：MCP Server 从队伍自己的配置读取题目服务器地址和 Team Agent Key，再把工具调用映射成 HTTP 请求。

HTTP API HWCTF_API_ENDPOINT
MCP 发现路径 /mcp 或 /api/v1/mcp
鉴权方式 X-Agent-Key

题目服务器保留 /mcp 和 /api/v1/mcp 作为发现入口；实际比赛建议由队伍在 Agent 容器内运行自己的 MCP 适配器，所有工具最终仍调用题目服务器 Agent API。

MCP 工具映射

工具名HTTP 请求说明
list_challengesGET /api/v1/agent/challenges列出当前模式下可做题目。
get_challengeGET /api/v1/agent/challenges/<challenge_id>读取题面、附件、环境和提示信息。
start_envPOST /api/v1/agent/challenges/<challenge_id>/env启动动态环境。
use_env_urlCONNECT env.urlWeb 直接访问 env.url；Pwn 连接固定 TCP 入口后先发送 routing_header。
stop_envDELETE /api/v1/agent/env/<env_id>关闭指定动态环境。
unlock_hintPOST /api/v1/agent/challenges/<challenge_id>/hints/<hint_id>/unlock解锁提示并记录扣分。
submit_flagPOST /api/v1/agent/challenges/<challenge_id>/submit提交 Flag，body 为 {"flag":"..."}。
get_scoreboardGET /api/v1/agent/scoreboard读取当前模式排行榜。

MCP 工具返回的错误应保留 HTTP API 的 error.code、error.message 和 error.description，便于 Agent 按错误码重试、换题或停止当前动态环境。

环境变量

HWCTF_API_ENDPOINT=http://challenge-server:9080/api/v1

这是推荐变量名，不是平台强制注入项。队伍需要在镜像内配置或默认启动逻辑中自行设置。

调用约束

  • MCP 工具失败时应把平台原始错误传回 Agent，便于判断是鉴权失败、题目锁定、环境冲突、频率限制还是 Flag 错误。
  • 不要在 MCP 日志、提示词、调试输出或仓库中打印完整 Team Agent Key。
  • 比赛模式下 MCP Server 应随提交的 Docker Agent 容器运行，不应依赖赛中人工交互。

题目配置说明

一、动态题目录

challenge-web-login/
├── challenge.yaml
├── Dockerfile
├── src/
├── images/
│   └── hwctf-web-login.tar.gz
├── attachments/
├── solution/
│   ├── writeup.md
│   ├── solve.py
│   └── exp.py
└── test.sh

images/ 可选但推荐。已经构建好的 x86_64 题目镜像放这里；平台上传题目包后会分发到在线题目服务器，由题目服务器自动 docker load 或 docker build。

二、必须准备的内容

题目名称展示给参赛队伍的题目名。 分类当前只使用 Web / Pwn。 分值基础分，平台按解出顺序和提示扣分计算最终得分。 Flag 数量默认 1；多 Flag 题必须全部提交正确后才算完成。 题目描述比赛时公开给 Agent 的题面。 提示只能配置一条提示，默认扣分比例为 20%。 Docker 镜像题目服务器宿主机能看到的镜像名。 镜像包可选；预构建的 docker save 产物，适合离线导入。 构建目录可选；不带镜像包时，使用 Dockerfile 在题目服务器本地构建。 容器端口题目服务在容器内监听的端口。 协议Web 用 http，Pwn 用 tcp。

三、动态 Flag

平台在每次启动队伍题目环境时生成动态 Flag，并通过环境变量注入容器。出题人只配置 Flag 数量，不需要在平台填写真实 Flag；题目程序从 FLAG、FLAG_1、FLAG_2 这类环境变量读取，不要把正式 Flag 写死在镜像里。

# Python 示例
import os

FLAG = os.environ.get("FLAG", "flag{local_test}")
FLAG_1 = os.environ.get("FLAG_1", FLAG)
FLAG_2 = os.environ.get("FLAG_2", "flag{local_test_2}")

六、交付方式 A：带镜像包

# 出题机或构建机
docker build -t hwctf/web-login:1.0 .
docker save hwctf/web-login:1.0 | gzip -9 > images/hwctf-web-login.tar.gz

# 在题目服务器宿主机加载镜像
docker load -i images/hwctf-web-login.tar.gz

推荐用于内网离线比赛。管理员把题目包上传到平台后，平台会分发到题目服务器，题目服务器导入时自动加载 image_package 指向的镜像包。

七、交付方式 B：本地构建

# 在题目服务器宿主机的题目目录执行
docker build -t hwctf/web-login:1.0 -f Dockerfile .

# 构建完成后确认镜像名与 challenge.yaml 中 env_image 一致
docker images | grep hwctf/web-login

仅适合题目服务器能访问基础镜像和构建依赖的环境。管理员上传题目包后，题目服务器会按 build.context 和 build.dockerfile 本地构建。

八、导入顺序

1. 准备 challenge.yaml、题目源码、镜像包和赛后材料
2. 单题可直接压成 zip/tar.gz；多题可一个包内放多个题目目录
3. 批量导入时，也可以一个大 zip 内放多个单题 zip
4. 在平台题目管理上传题目包，平台会分发到所有在线题目服务器
5. 导入完成后，在题目列表确认每题是否“带镜像包 / 本地构建 / 远程拉取”
6. 使用测试队伍 Agent 启动环境、查看提示、提交正确/错误 Flag 验证

九、赛后材料

solution/ 下的 writeup、poc、exp 只作为赛后归档和验题材料，不会在比赛期间通过平台暴露给参赛队伍。
